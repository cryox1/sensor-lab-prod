# Hardware — AS3935 storm node (`storm01`, BME280 optional)

**Status:** R1 Proposed (firmware written and compile-tested; hardware not yet
assembled — flip R1 to Implemented after the node is built and verified),
R2 Implemented
**Scope:** A storm node on a FireBeetle 2 ESP32-C6: DFRobot Gravity AS3935
lightning sensor (SEN0290, <https://wiki.dfrobot.com/sen0290/>) plus an
optional GY-BME280 (tolerated absent — the current build is SEN0290-only).
Current placement is **indoors** (`AS3935_LOCATION 0`), powered by USB or a
LiPo at the BAT connector; the original outdoor-solar option (DFR0579 Solar
Power Manager Micro,
<https://wiki.dfrobot.com/Solar_Power_Manager_Micro_SKU__DFR0579>, with a
LiPo buffer) remains valid. Sketch
`firmware/bme280_as3935_firebeetle2_c6_deepsleep/`, device id `storm01`,
static IP `10.0.0.72` (`.71` belongs to the CYD wall panel). Board contract:
`hw-firebeetle2-c6.md`.

## Behavior (planned / as-coded)

- **Power:** indoors, USB or LiPo at the BAT connector; `batt_v` is published
  via the onboard VBAT divider. For the optional solar variant: solar panel →
  DFR0579 (MPPT charger, 0.5–4 V panels, max ~70 mA charge) → LiPo → the
  FireBeetle's **own BAT connector**. The DFR0579 is only the charger — its
  3.3 V/90 mA output cannot supply WiFi TX peaks (200–350 mA), which come
  from the battery via the FireBeetle's regulator. On a solar budget the
  SEN0290's red power LED (~1–2 mA continuous) must be removed/cut — it would
  be a top consumer in the harvest budget; indoors on USB it doesn't matter.
- **Why deep sleep works here (unlike the ENS160 nodes):** the AS3935 detects
  lightning autonomously at ~µA while permanently powered; the ESP only needs
  to wake when there is something to say. The sensor keeps its configuration
  and calibration through ESP deep sleep — it is configured once per cold
  boot (`manualCal`: indoors (`AS3935_LOCATION 0`), disturber detection on,
  tuning caps 96 pF, I2C addr 0x03 via DIP switches).
- **Two wake sources:** the RTC timer (`SLEEP_SECONDS 600` — wider than the
  5-min nodes because the ~0.3 W panel budget is tight in winter) for a
  climate heartbeat, and a **GPIO deep-sleep wake from the AS3935 IRQ pin**
  on GPIO2 (must be an LP GPIO 0–7; GPIO0 is the VBAT divider).
- **Burst dampening:** after a published strike, further strikes within
  `LIGHTNING_COOLDOWN_S 60` are only accumulated in RTC memory (count, min
  storm-front distance, max raw energy) at ~ms wake cost — no WiFi — and are
  folded into the next publish. Disturber/noise IRQs never publish; a flood
  guard masks the disturber IRQ after 100 disturber wakes without a publish.
- **Clock:** no simulated fixed-step epoch (lightning wakes at arbitrary
  times would corrupt it); the C6 RTC carries system time through deep sleep,
  NTP on cold boot / every 24 publishes / when invalid.
- **Published fields:** `batt_v` and `lightning_count` (strikes since last
  publish, 0 when quiet — the live "listening" baseline; suppressed if the
  AS3935 doesn't answer on I2C so a dead sensor can't masquerade as a quiet
  sky) on every publish; `lightning_km` (min storm-front distance, km, 1 =
  overhead, 63 = out of range) and `lightning_energy` (max raw intensity,
  21-bit, unitless) only when strikes are pending; `temp_c`, `pressure_hpa`,
  `humidity`, `heat_index_c` only when a BME280 is fitted. Without the
  BME280 the heartbeat still publishes (batt_v + count). BME280 over BME680
  by choice: outdoor `gas_kohm` is noise, and pressure fits a storm node.
- **Diagnostics:** retained debug record on `sensors/lab/storm01/debug`
  with wake cause, strike/disturber/noise counters, and the flood-guard state.

## Requirements

### R1 — Solar storm node (Proposed)

Field the BME280+AS3935 pair as a self-sufficient outdoor node: solar keeps
the LiPo charged, the LiPo buffers dark days, lightning events publish
promptly, climate publishes on the heartbeat.

Acceptance criteria:

- **Given** the assembled node, **when** the 600 s timer fires, **then** one
  telemetry message with the climate fields + `batt_v` arrives on
  `sensors/lab/storm01/telemetry`.
- **Given** a lightning strike (or a functional test with a lighter/spark
  generator near the antenna), **then** a publish with `lightning_km` /
  `lightning_energy` / `lightning_count` arrives promptly — and follow-up
  strikes within 60 s are accumulated, not published individually.
- **Given** a disturber-heavy environment, **then** the node does not drain
  itself: disturber wakes stay off WiFi and the flood guard engages (visible
  in the debug record).
- **Given** a week of mixed weather, **then** `batt_v` stays in a healthy
  band (charge ≥ consumption) — the core solar viability check.

### R2 — Lightning fields end-to-end (Implemented)

Carry `lightning_km`, `lightning_energy`, `lightning_count` through the
stack, mirroring how `gas_kohm` was added (`hw-bme680.md` R1): columns in
`postgres/init.sql` + ingest `MIGRATIONS`/`INSERT`/`to_row`, API projections
(incl. the lifespan self-migration and `THRESHOLD_METRICS`), web `METRICS`
entries + a SensorCard row.

- Column types: `lightning_km SMALLINT` (6-bit register, 1–63),
  `lightning_energy INTEGER` (21-bit raw, max 2 097 151),
  `lightning_count INTEGER` (uint16 accumulator, up to 65 535).
- **History bucketing is NOT plain AVG** (unlike the scalar metrics; second
  exception besides GPS): `SUM(lightning_count)` (per-publish delta → storm
  totals), `MIN(lightning_km)` (closest approach), `MAX(lightning_energy)`
  (peak intensity) — mirroring the firmware's own burst folding.

Acceptance criteria:

- **Given** a publish with the three lightning fields, **then** they are
  stored on the `telemetry` row and returned by `/latest`, `/history` (raw
  and bucketed), `/history-all`, and the CSV export.
- **Given** two publishes in one bucket (counts 3 and 2, distances 12 and 8,
  energies 174000 and 90000), **then** the bucket reports count 5, km 8,
  energy 174000.
- **Given** a quiet interval, **then** heartbeat rows carry
  `lightning_count = 0` (not NULL), so the strikes chart shows a baseline.

## Non-goals

- No always-on operation — the whole point is the µA listening budget
  (contrast: `hw-ens160.md`).
- No per-strike storm history in the first cut (strikes within a cooldown
  window are aggregated; individual-strike fidelity is a non-goal).
- No AS3935 antenna auto-tuning — fixed 96 pF (DFRobot's calibrated default
  for the SEN0290).

## Open questions

- Indoor disturber/noise rates: if the debug record shows floods or serial
  spams "noise level too high", raise `setNoiseFloorLvl` /
  `setWatchdogThreshold` / `setSpikeRejection` (defaults 2/2/2, range 0–7).
- (Solar variant only) winter viability of the 600 s heartbeat with the
  ~0.3 W panel — may need widening (or a bigger panel) after the first
  `batt_v` trend data.
